# goldenwave v0.2 — 摄入路由器 PRD

> 状态：草案｜版本：v0.2 目标特性｜作者：张金波｜日期：2026-06-30
> 关联：SPEC.md L3-workflow｜这是 goldenwave 从"模板"跃迁为"系统"的关键一跃。

## 1. 背景与目标

v0.1 给了用户一套规范的**空骨架**（init 后得到目录 + SPEC）。但"往哪填、怎么填"仍全靠人，与"又一个 Obsidian 模板"区分度不足。

**路由器要解决的核心问题**：用户把任意原料丢进 `inbox/`，系统**自动判断它属于哪一层、渲染成合规页面、并安全写入**——把 goldenwave 从"结构标准"变成"会自己长的知识系统"。

**北极星**：inbox 里的一条原料，经路由器处理后，正确落到 事实(L1)/知识(L2)/人格(persona) 的合规页面，且全程可审计、可回退。

## 2. 设计原则（不破 SPEC 红线）

1. **半自动，不全自动**：agent 判断 + 人确认。高敏/低稳定一律先进草稿。
2. **可审计**：每步产物皆 markdown，路由决策写进 log，可 git diff、可 revert。
3. **三维分流**：严格区分 事实/知识/人格，不混淆。
4. **不确定就问**：路由置信度低 → 进 `inbox/_pending/` 待人裁决，绝不静默入库。

## 3. 流水线（借 OpenHuman capture→score→render→inject）

```
inbox/原料
  │
  ├─① capture   收集 + 标准化（读文件/链接/文本，提取纯文本 + 元数据）
  ├─② route     分类器：事实(L1域) / 知识(L2类型) / 人格(persona facet) / 丢弃
  ├─③ score     证据分级(verbatim>artifact>impression) + 稳定度 + 敏感度
  ├─④ render    按目标层 frontmatter 规范渲染成候选页
  └─⑤ inject    写入 or 进草稿；更新索引 + 双链 + log
```

### 各步细节

**① capture**：支持 .md/.txt/链接/对话片段。输出统一中间格式 `{text, source, captured_at}`。

**② route（核心分类器）**：一个 agent prompt，对每条原料判定：
- 维度：是"此刻为真的事实" → L1（再判九域）；"可复用知识" → L2（再判八类）；"我怎么想/说话" → persona。
- 输出 `{layer, target, confidence, reason}`。
- confidence < 阈值 → `_pending/`。

**③ score**：
- 证据分级：原话=verbatim，成稿=artifact，他人转述=impression。
- 稳定度：一次性 vs 长期；决定 refresh 周期。
- 敏感度：L0–L3，决定是否进 git / 是否脱敏。

**④ render**：套用 templates/ 对应模板，自动填 frontmatter（id/type/tags/verified/sensitivity…）。

**⑤ inject**：
- 高稳定 + 低敏 + 高置信 → 直接写入对应目录。
- 否则 → `inbox/_pending/` 草稿，附路由决策卡，等用户 confirm。
- 写入后：更新 `wiki/<type>/index.md` 或 `profile/<域>/index.md`、补双链、追加 `.kb/log.md`。

## 4. 交付物

- `skills/goldenwave-ingest/SKILL.md` — 路由器 skill（挂 agent 调用）
- `scripts/route.py` — 可选的批处理 CLI（扫 inbox → 产候选）
- `inbox/_pending/` 草稿区约定 + 决策卡格式
- SPEC L3 从"预留"改为"已实现"

## 5. 验收标准

1. 投 10 条混合原料（含事实/知识/人格/敏感），路由准确率 ≥ 80%，敏感项 100% 进草稿不直接入库。
2. 每次路由在 `.kb/log.md` 留可读决策记录。
3. 错误路由可一条命令 revert（git）。
4. 不依赖任何云服务即可跑（本地模型亦可）。

## 6. 非目标（划给 v0.3+）
